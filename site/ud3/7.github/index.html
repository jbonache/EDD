    7. Repositoris remots amb Git. Github.

    Introducció

    Com ja hem comentat, Git és un sistema de control de versions distribuït, de manera que el repositori es troba físicament distribuït entre diferents equips.

    En l'apartat anterior, hem vist com treballar amb Git de forma local, inicialitzant repositoris, afegint continguts, i creant diferents branques sobre les quals treballar amb codi inestable o de proves.

    L'actual mercat de treball s'ha descentralitzat notablement, donant lloc a equips de treball deslocalitzats i fent necessaris certs mecanismes de coordinació i organització del treball en aquests equips.

    En aquest apartat abordarem el treball amb Git de forma distribuïda, per poder accedir a repositoris remots, i poder combinar el nostre treball amb el d'altres desenvolupadors del nostre equip.

    D'altra banda, introduirem la plataforma de desenvolupament col·laboratiu Github, que ens permet allotjar els nostres projectes i gestionar el control de versions amb Git. La plataforma, creada el 2008, s'utilitza principalment per allotjar projectes de programari lliure de forma pública, i el 2018 va ser adquirida per Microsoft. Actualment, Github compta amb més de 100 milions d'usuaris, i més de 420 milions de projectes.

    A més del codi i el control de versions per als projectes, ofereix altres serveis, com una wiki, pàgina web, eines gràfiques i col·laboratives o l'IDE al núvol Codespaces, per interactuar amb els repositoris, entre altres.

    Repositoris remots

    Per ara hem vist com crear i gestionar repositoris amb Git, així com treballar amb diferents branques, sempre de forma local. Quan treballem en un equip deslocalitzat, necessitem compartir aquest treball local. És ací on entra en acció l'arquitectura distribuïda de Git.

    Com ja sabem, Git és un SCV distribuït. Això significa que el repositori complet pot estar ubicat en diferents ordinadors, en lloc de centralitzar-se en un únic servidor. No obstant això, això no significa que no es requerisca d'un servidor que s'encarregue de mantenir el que seria el repositori principal.

    Directoris de treball i repositoris de tipus bare

    En els exemples pràctics anteriors hem utilitzat l'ordre git init per inicialitzar un directori com a directori de treball en el qual poder realitzar un control de versions de forma local. En aquest directori de treball, Git genera en la pròpia arrel el directori ocult .git, dins del qual s'ubica tota la informació necessària per al control de versions de forma local.

    Quan diversos desenvolupadors necessiten compartir el seu treball, ho faran a través d'un repositori de tipus bare, la traducció del qual podria ser repositori descobert, buit o net, i que s'ubicarà en un servidor centralitzat. Els equips de treball, utilitzaran una còpia d'aquest repositori per treballar de forma local, i sincronitzaran el seu treball amb el repositori central.

    Aquesta arquitectura podria veure's de la següent manera:

    Aquesta segueix sent una arquitectura distribuïda, ja que el repositori es troba en els diferents equips de treball, en forma de repositori local, però a més, existeix un servidor que emmagatzema el repositori descobert. Aquest repositori no tindrà directori de treball, de manera que el seu contingut tindrà la mateixa estructura que el contingut del directori .git dels repositoris locals.

    El flux de treball en aquests casos serà el següent:

    1. Creació del repositori de tipus bare en el servidor. (git init --bare)
    2. Clonació del repositori en cada equip de treball. (git clone adreça_del_repositori)
    3. Treball en la còpia local del repositori i el directori de treball. Aci s'afegeixen o eliminen recursos (add/rm), es confirmen en el repositori local (commit), etcètera.
    4. Sincronització entre el repositori local i el del servidor. Quan s'envien dades al servidor es realitza una operació de PUSH (git push), i quan desitgem obtenir dades del servidor s'utilitza FETCH (git fetch) i PULL (git pull).

    Profunditzarem en aquest flux de treball de forma pràctica en el cas pràctic següent.

    Github

    Quan desitgem crear els nostres projectes per treballar en equip, o simplement per disposar d'un repositori centralitzat a Internet amb el qual portar el control de versions dels nostres projectes mitjançant Git, Internet ens ofereix un gran ventall de possibilitats.

    Potser la més popular d'elles sigui Github, encara que des de la compra d'aquesta per part de Microsoft i les reticències que aquest fet va generar en la comunitat, han crescut altres plataformes com Gitlab o Bitbucket.

    Així i tot, Github compta amb una creixent base d'usuaris i projectes (més de 100 milions d'usuaris i més de 430 milions de projectes).

    Github es defineix com una plataforma de desenvolupament col·laboratiu, per la qual cosa a més de l'allotjament i el control de versions mitjançant Git, també ens ofereix eines per a la gestió de projectes i el treball en equip, com una wiki o webs de documentació, entre moltes altres.
