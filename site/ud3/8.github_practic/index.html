    8. Treballant en remot amb Git i Github.

    Repositoris remots

    Anteriorment hem vist com crear un repositori a partir d'un directori de treball en la màquina local.

    Ara veurem com obtenir un repositori ubicat en una altra màquina. A aquesta operació se li coneix com a clonació d'un repositori.

    Quan parlem de clonar un repositori ens referim a portar al nostre equip un espai de treball remot complet, incloses totes les confirmacions i branques realitzades en l'altre equip.

    Encara que Git treballa de forma distribuïda, necessitarem un repositori de referència, que faça de servidor.

    Pas 1. Creació del servidor

    Quan utilitzem git init, establim el directori actual com a directori de treball. Això implica la creació d'un subdirectori ocult .git amb tota la informació sobre el control de les versions. Això ens serveix per treballar localment, creant els nostres commits, branques, etc. però no ens serveix d'aquesta manera per treballar de forma distribuïda.

    Perquè en el directori actual es genere un repositori que puguem compartir, necessitem el que es coneix com un repositori descobert (bare), que tindrà una estructura diferent als directoris de treball, ja que prescindirà de la carpeta .git i utilitzarà el propi directori base per guardar el seu contingut.

    En primer lloc, creem el directori en el qual desitgem crear el repositori:

    $ mkdir projecteCompartit

    Una vegada creat, accedim al directori i inicialitzem el repositori descobert:

    $ cd projecteCompartit

$ git init --bare

consell: S'està utilitzant «master» com a nom de la branca inicial. Aquest nom de branca
consell: per defecte es pot canviar. Per a configurar el nom inicial de la branca que
consell: s'utilitzarà en tots els repositoris nous, i que suprimirà aquest avís, useu:
consell: 
consell:        git config --global init.defaultBranch <nom>
consell: 
consell: Els noms més usats habitualment en lloc de «master» són «main», «trunk» i
consell: «development». La branca acabada de crear es pot canviar de nom amb l'ordre:
consell: 
consell:        git branch -m <nom>

S'ha inicialitzat un repositori buit del Git en /tmp/projecteCompartit/

    Nom de la branca inicial

    Tal i com ens indica a l'avís, podem canviar el nom predeterminat de les branques inicials configurant el paràmetre de git init.defaultBranch.

    Si ja hem creat el repositori i volem canviar el nom d'aquesta branca, ho fem amb git branch -m nom. Si pe rexemple volem utilitzar main en lloc de master, tal i com fa github, podem fer:

    $ git branch -m main

    Si consultem el contingut del directori, veurem que té la mateixa estructura que el directori .git en un repositori local.

    $ ls
branches config description HEAD hooks info objects refs

    Pas 2. Clonació del repositori (en l'ordinador local)

    En general, per clonar un projecte farem:

    git clone origen [destí]

    D'aquesta manera, si per exemple desitgem clonar un repositori que es troba en una carpeta del nostre propi equip, farem, per exemple:

    $ git clone ~/scv/projecteCompartit ~/scv/projecte3

S'està clonant a «/home/dam/scv/projecte3»...
warning: Sembla que heu clonat un repositori buit.
fet.

    Amb això hem clonat el projecte projecteCompartit en una carpeta anomenada projecte3. Si ometem aquesta segona carpeta, el repositori es clonarà en el directori actual amb el mateix nom que la carpeta original.

    D'altra banda, si el que volem és clonar un repositori remot, el que farem serà connectar-nos a ell, bé a través dels protocols HTTPS, o SSH.

    Suposem que l'ordinador on hem creat el repositori compartit té l'adreça IP 192.168.1.132. En aquest cas hauriem d'utilitzar l'adreça IP de l'ordinador al qual ens volem connectar. Podem utilitzar fins i tot nostra pròpia màquina consultant la nostra adreça IP o directament utilitzant l'adreça IP local de l'equip 127.0.0.1.

    Ens connectaríem a ell amb:

    $ git clone ssh://dam@192.168.1.132/home/dam/scv/projecteCompartit

Cloning into 'projecteCompartit'...
dam@192.168.1.132's password: 
warning: You appear to have cloned an empty repository.

    Observa que no hem indicat una ruta de destí, amb la qual cosa es clonarà el repositori compartit en una carpeta amb el mateix nom en el directori actual.

    Això ens demanarà la contrasenya de l'usuari amb el qual ens hem connectat per SSH. Si no desitgem utilitzar el mateix usuari que en local, podríem crear un nou usuari per realitzar aquestes tasques.

    Finalment, el sistema ens avisa que hem clonat un repositori buit. Això es deu al fet que el repositori ha estat inicialitzat, però encara no disposa de contingut.

    Pas 3. Afegint contingut al repositori

    Ara, des de l'ordinador local, ens situem en la carpeta on hàgem clonat el repositori:

    $ cd projecteCompartit/

    Afegim contingut, per exemple un fitxer anomenat fitxer1.md amb el contingut “# Projecte Compartit”:

    $ echo "# Projecte Compartit" > fitxer1.md

    L'afegim per confirmar en el següent commit:

    $ git add fitxer1.md

    I fem la confirmació:

    $ git commit -m "Afegit fitxer1.md"

[master (root-commit) c76e80e] Afegit fitxer1.md
 1 file changed, 1 insertion(+)
 create mode 100644 fitxer1.md

    Fixa't que ens mostra el commit [master (root-commit) c76e80e] ([main (comissió arrel) 8fbbb4c]), indicant que es tracta de la branca master (o main si hem canviat el nom), i a més és el commit arrel. Això està relacionat amb l'avís de repositori buit que ens ha mostrat quan hem clonat el repositori, i indica que es tracta del primer commit que es fa en ell.

    Si ara comprovem l'estat del repositori: 

    $ git status

On branch master
Your branch is based on 'origin/master', but the upstream is gone.
 (use "git branch --unset-upstream" to fixup)
nothing to commit, working tree clean

    O bé:

    En la branca main
La vostra branca està basada en «origin/main», però la font no hi és.
  (useu «git branch --unset-upstream» per a arreglar-ho)

no hi ha res a cometre, l'arbre de treball està net

    Ens indica que ens trobem en la branca master/main, però que aquesta no existeix en upstream (l'origen de la clonació). En aquesta situació, això és normal, ja que no hem sincronitzat encara aquest primer commit en el servidor.

    Pas 4. Enviant el commit al servidor

    Els canvis realitzats s'han fet en la nostra còpia local del projecte, per la qual cosa si volem compartir-los amb la resta de col·laboradors del projecte, haurem de sincronitzar-la amb el servidor. Per enviar aquests canvis al servidor, utilitzarem l'ordre git push:

    $ git push origin master

    Les opcions origin i master/main poden eliminar-se, deixant únicament l'ordre com git push. Si no indiquem el servidor remote, git escollirà el que tinguem configurat, que en aquest cas és origin. Podem comprovar quin o quins són els nostres servidors remots amb:

    $ git remote

    Si ara comprovem de nou l'estat:

    $ git status

    o bé:

     git status

    Pas 5. Portar contingut del servidor

    Quan desitgem obtenir les modificacions que han fet sobre el projecte altres desenvolupadors, utilitzarem les ordres de git fetch i git pull.

    Per a això, partirem de les modificacions realitzades en el repositori en l'exercici anterior.

    • git fetch [nom-remot]: Permet obtenir els canvis realitzats en el projecte remot des que el vam clonar o des que vam obtenir els canvis per última vegada. Aquest comandament únicament porta les dades, però no les combina amb el nostre directori de treball. Serveix per veure els canvis, i aplicar-los manualment quan es cregui convenient.
    • git pull [nom-remot]: Permet obtenir els canvis realitzats en el projecte des que el vam clonar o des que vam obtenir els canvis per última vegada, i a més, combina aquests amb el nostre directori de treball. Aquest flux de treball serà bastant més còmode i ràpid que utilitzar fetch, encara que treu una mica de control sobre el repositori.

    Github

    La plataforma de desenvolupament col·laboratiu Github ens permet crear repositoris git centralitzats i compartits de tipus descobert (bare).

    En aquest apartat anem a crear un compte en aquesta plataforma i crearem el nostre primer repositori públic.

    Creació d'un compte

    En primer lloc, accedim al lloc web de GitHub i creem un compte gratuït. Per a això, no hem de fer més que introduir la nostra adreça de correu on se'ns demana i fer clic en el botó de Sign up for Github (Compte, les següents captures poden estar desactualitzades, però el procediment en general és el mateix).

    S'iniciarà ara un petit assistent en el qual confirmem o introduïm el nostre correu si no ho havíem fet, escollim una contrasenya, un nom d'usuari, diem si volem rebre actualitzacions, i finalment verifiquem el compte resolent un petit trencaclosques.

    Acabat aquest assistent, fem clic en Create Account. En aquest moment, se'ns enviarà al correu indicat un codi de verificació, per completar la creació del compte.

    Fet això, ja ens apareixerà la pantalla inicial de Github, presentant diverses opcions:

    Des d'ací podríem crear ja un repositori, seleccionant l'opció Create a Repository. De moment, farem clic en Skip this for now per anar a la pantalla principal, des de la qual tindrem accés a totes les opcions que ens ofereix Github.

    D'aquesta pantalla principal de Github anem a destacar tres zones:

    • 1. En la columna central, veurem tota l'activitat recent, tant pròpia com dels nostres contactes.
    • 2. La columna de l'esquerra ens mostrarà algunes dades sobre els nostres repositoris, equips, etc. Des d'ací podem començar a crear repositoris.

    • 3. En la part superior, disposem d'una barra de menú, amb un enllaç a la pàgina principal, un quadre de cerca, un menú amb quatre entrades:

      • Pull Requests: Per accedir i validar les peticions de fusió de codi d'altres usuaris sobre els nostres repositoris,
      • Issues: Per gestionar anotacions i suggeriments que es fan sobre el nostre repositori, ja sigui per corregir errors, peticions de noves funcionalitats o preguntes sobre el funcionament entre altres.
      • Marketplace: Per afegir extensions i altres funcionalitats a Github.
      • Explore: Per explorar repositoris que Github considera que poden ser del nostre interès.

      A més, en aquesta barra superior, disposem de tres icones:

      • La campana, que ens mostra si tenim alguna notificació,
      • La icona d'Afegeix (signe “+”), que ens serveix per gestionar nous repositoris, organitzacions o projectes. En un projecte, a part de la gestió del repositori tindrem altres opcions relacionades amb la gestió de projectes, com pugui ser la gestió de tasques.
      • La icona amb el nostre avatar, sobre el qual podem controlar diversos aspectes relacionats amb el nostre compte (perfil, repositoris, equips, etcètera).

    Creació d'un Token d'Accés Personal

    Una vegada creat el nostre compte en Github i abans de començar a crear un repositori, anem a crear un token d'accés personal (PAT, Personal Access Token).

    Quan treballem des de la línia d'ordres amb Git sobre els nostres repositoris emmagatzemats en Github, necessitarem algun mode d'autenticar-nos en la pròpia plataforma. Tradicionalment, ha estat suficient amb indicar, en el moment de publicar els nostres canvis en el servidor, les nostres credencials. És a dir, el nostre usuari en Github i la nostra contrasenya.

    Amb la finalitat d'incrementar la seguretat en la plataforma, des del 13 d'agost de 2021, Github no permet aquest tipus d'autenticació des de la línia d'ordres, i requereix fer-ho mitjançant HTTPS amb un token d'accés generat per la pròpia plataforma.

    Anem a veure doncs, com generar aquest token, que haurem d'emmagatzemar en el nostre equip i utilitzar quan se'ns demani una contrasenya.

    • Pas 1. Localitzem la icona amb la nostra foto de perfil en la part superior dreta, i fem clic en ella per accedir a la nostra configuració de Github (Settings).
    • Pas 2. Fem clic en l'opció de paràmetres del desenvolupador (Developer Settings) en el panell de l'esquerra.
    • Pas 3. En la pàgina de paràmetres del desenvolupador, en el panell de l'esquerra fem clic en Personal Access Tokens.
    • Pas 4. En la secció de Personal Access Tokens seleccionem Generate New Token.
    • Pas 5. Indiquem una descripció per al nostre token, per exemple “Token per a l'accés per terminal”, així com l'àmbit o abast d'aquest token. Això determinarà què es podrà i què no es podrà fer amb aquest token. En el nostre cas, anem a seleccionar totes les opcions d'interacció amb el repositori (repo, admin:repo_hook i delete_repo), i fem clic en Generate Token.
    • Pas 6. Se'ns generarà un token que haurem de copiar i conservar per utilitzar com a contrasenya d'accés per terminal. Per copiar-lo podem fer clic en la icona de còpia que es troba a la seva dreta. Tingueu especial precaució amb ell i guardeu-lo bé, ja que per qüestions de seguretat, aquest token no se us tornarà a mostrar una vegada eixiu de la pàgina.

    Creació d'un repositori

    Anem a crear ara el nostre primer repositori en Github. Per a això, des de la finestra inicial, seguirem els següents passos.

    Pas 1. Create Repository En primer lloc, bé des de la secció Create your First Project de la secció de l'esquerra, o bé des de la primera entrada del submenú d'Afegeix (+), fem clic en Create Repository.

    Això ens portarà a la pàgina de creació del nou repositori.

    Pas 2. Emplenant les dades del repositori

    Per crear el repositori haurem de facilitar certa informació sobre aquest:

    • El nom,
    • Una descripció,
    • Si serà públic o privat,
    • Opcions d'inicialització del repositori, com crear un fitxer README.md, un .gitignore, o una llicència.

    En el nostre cas, hem afegit les següents dades:

    Fixa't en certs detalls:

    • El nom del repositori està vinculat a l'usuari. D'aquesta manera, diferents usuaris podran tenir repositoris amb el mateix nom, i es facilita també la importació de repositoris d'altres usuaris. Aquest tindrà doncs un format tipus nomDeUsuari/Repositori, en el nostre cas, AlumnoDAM-MGH/mi-primer-repo. En el vostre cas, utilitzareu elTeuNomDUsuari/el-meu-primer-repo.
    • Hem seleccionat que s'inicialitzi el repositori amb un fitxer README, però sense .gitignore ni llicència. Aquest pas podem ometre'l, en aquest cas, hauríem d'inicialitzar el repositori manualment. Si el deixem sense inicialitzar, Github ens indicarà quins passos hem de realitzar des de la terminal per completar la inicialització.
    • D'altra banda, ens indica que la branca per defecte generada és main. Si recordeu, en git, aquesta branca principal per defecte es deia master, però es podia canviar. Inicialment, en Github també era així, però des d'octubre de 2020, s'utilitza main en lloc de master.

    Pas 3. Creació del repositori

    Una vegada tenim les dades emplenades, fem clic en Create Repository, amb el que ja tindrem llest el nostre repositori.

    Els repositoris que creem disposaran també d'una web des de la qual podrem consultar el codi del repositori, gestionar-lo i modificar-lo:

    En aquesta pàgina tenim diversos elements d'interès:

    • En la part superior esquerra tenim la ruta al nostre repositori. Si ens trobem en qualsevol subcarpeta d'aquest, ens la mostrarà ací, donant la possibilitat de retrocedir en l'estructura de directoris.
    • A la part superior dreta, tenim informació sobre el nombre d'usuaris que estan pendents del nostre projecte (Watch/Unwatch), els que l'han valorat positivament (Star) o les bifurcacions que altres usuaris hagin fet del projecte (Fork). Des d'aquests botons, també podem seguir o deixar de seguir el projecte, afegir-li estrelles o crear un fork.
    • Després tenim un menú amb les diferents opcions del projecte: Explorar el codi (code), els assumptes pendents (issues), les peticions de fusió de codi (pull requests), la Wiki, o els ajustos (Settings) del projecte, entre altres.
    • Posteriorment tenim diversos botons, entre els quals podrem canviar entre branques (ara se'ns mostra la branca Main), afegir fitxers, o bé obtenir el codi del repositori (Code).
    • En la part central, i amb major grandària, observem el contingut del nostre repositori, juntament amb informació relativa als commits realitzats o el HEAD. Com podeu comprovar, també ens mostra el contingut del fitxer README.md. En general, en qualsevol carpeta que tinguem, podem afegir un fitxer README.md, que Github interpreta que és el document de descripció bé del projecte o repositori, o bé de la carpeta en la qual estiguem.

    Pas 4. Clonant el repositori Una vegada creat aquest repositori, anem a descarregar-lo en el nostre equip. Per a això, busquem el botó verd amb el text Code, i despleguem el submenú amb les diferents opcions de clonació:

    Des d'ací, podem clonar el repositori bé a través de HTTPS, SSH, o la interfície de línia d'ordres (CLI) de Github. També podem descarregar el contingut del projecte en format ZIP, sense informació sobre el control de versions.

    En el nostre cas utilitzarem la primera opció, per realitzar la clonació per HTTPS.

    Així doncs, seleccionarem el text https://github.com/AlumnoDAM-MGH/mi-primer-repo.git i el copiarem, o bé podem fer-ho directament amb el botó de copiar situat a la seva dreta.

    Branques i forks

    Recordem que Github és una de les majors plataformes d'allotjament de projectes basats en programari lliure. Una de les premisses d'aquest tipus de projectes és la possibilitat de compartir el codi, modificar-lo i adaptar-lo a les nostres necessitats.

    Github ens brinda la possibilitat de crear còpies completes d'un projecte al nostre compte d'usuari. Això es coneix com a fork, i permet adaptar projectes a les nostres necessitats, o crear projectes nous basant-nos en un ja existent.

    Aquest concepte de fork és diferent del de crear una branca, ja que genera una còpia física completa del repositori, amb la intenció de començar un desenvolupament independent a l'original. En canvi, una branca suposa una bifurcació, però dins del repositori original, amb la intenció de realitzar canvis que després s'aplequin a la branca principal.

    No obstant això, mitjançant un fork, també podrem col·laborar amb el projecte original, encara que no siguem membres del seu equip de desenvolupament. Quan creiem que els canvis que hem realitzat poden ser positius, o resolen algun error del codi original, podem fer un pull request al projecte original perquè els desenvolupadors considerin combinar el seu codi amb el nostre.
