    2. Junit Jupiter

    Pràctica amb JUnit Jupiter

    JUnit Jupiter és un framework de testeig per a Java. Quan creem un projecte amb Gradle, per exemple, tenim la possibilitat ja d'incorporar el framwork al projecte.

    Anem a veure com fer-ho.

    Pas 1. Configuració de l'entorn

    1. En el teu directori de treball, crea una carpeta per a un projecte Gradle nou (per exemple calculadora).
    2. Crea un nou projecte amb gradle init, amb les següents característiques concretes:
      • Projecte de tipus aplicació
      • Llenguatge Java 21
      • Estructur de projecte únic (Single application Project)
      • DSL Groovy
      • Framework de testing: JUnit Jupiter
      • Sense incorporar APIs i comportament experimental
    3. Edita el projete amb VSCode, i comprova el següent

      • Consulta el fitxer app/build.gradle i verifica que tens les dependències:

        // Use JUnit Jupiter for testing.
    testImplementation libs.junit.jupiter
    testRuntimeOnly 'org.junit.platform:junit-platform-launcher'

    testImplementationi testRuntimeOnly

    • testImplementation: especifica que JUnit Jupiter serà utilitzada en temps de compilació per a les proves. És a dir, s'inclou en el procés de compilació i es pot utilitzar per escriure les proves dins del projecte.

    • testRuntimeOnly: Aquesta dependència especifica que el JUnit Platform Launcher només es necessita en temps d'execució, no en compilació. És a dir, aquest component només s'utilitza per executar les proves (no per escriure-les), i no és necessari per a la compilació del codi.

    • Comprova que també apareix la tasca test:

    tasks.named('test') {
    // Use JUnit Platform for unit tests.
    useJUnitPlatform()
}

    Amb la qual configurem l'execució de les proves unitàries perquè utilitzin el JUnit Platform. Aquesta serà doncs la tasca responsable d'executar les proves. JUnit Platform és l'entorn que gestiona l'execució de les proves, el qual permet l'ús de funcionalitats modernes de JUnit (com anotacions tipus @Test, etc.)

    Pas 2. Creació del codi i les proves

    Com veuràs al build.gradle tenim definida la classe principal de l'aplicació mainClass = 'org.example.App'. Això implica que tenim definit el paquet org.example i l'estrucutra de carpetes següent:

    app/src
    ├── main
    │   ├── java
    │   │   └── org
    │   │       └── example
    │   │           └── App.java
    │   └── resources
    └── test
        ├── java
        │   └── org
        │       └── example
        │           └── AppTest.java
        └── resources

    Si llancem el projecte amb gradle run ontindrem un missatge Hello World. Observem el codi App.java:

    package org.example;

public class App {
    public String getGreeting() {
        return "Hello World!";
    }

    public static void main(String[] args) {
        System.out.println(new App().getGreeting());
    }
}

    Com veiem, el main crea una instància de la classe App i fa ús del mètode getGreeting d'aquesta, el qual mostra el missatge d'Hola món.

    Veiem ara el fitxer AppTest.java generat automàticament:

    /*
 * This source file was generated by the Gradle 'init' task
 */
package org.example;

import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class AppTest {
    @Test void appHasAGreeting() {
        App classUnderTest = new App();
        assertNotNull(classUnderTest.getGreeting(), "app should have a greeting");
    }
}

    Com veiem, aquest codi fa dos imports de Junit Jupiter:

    • import org.junit.jupiter.api.Test;, amb la qual importa l'anotació @Test de JUnit Jupiter, amb la qual marquem un mètode com a mètode de prova.
    • import static org.junit.jupiter.api.Assertions.*;: amb la qual importem les asercions de JUnit, que es poden utilitzar per verificar que el resultat d'una prova siga correcte (com assertNotNull). Una asserció serà doncs l'expressió que utilitzarem per validar els resultats.

    Aleshores, com veiem, el mètode appHasAGreeting està marcat amb l'anotació @Test, indicant que és una prova que s'ha d'executar. Aquesta serà una prova unitària, ja que provem una classe concreta.

    Dins aquest mètode, creem una instància (classUnderTest) de la nostra classe App i fem ús de l'asserció assertNotNull(classUnderTest.getGreeting()), amb la qual es comprova que el valor retornat per getGreeting() no siga nul. Si el missatge existeix, la prova passarà. Si no, fallarà, mostrant el missatge "app should have a greeting".

    Per tal d'executar aqeusta prova, només caldrà executar la tasca test amb gradle test.

    Quan ho fem, GRadle buscarà la classe amb l'anotació @Test i executarà els seus mètodes corresponents. Si la prova passa, el resultat serà semblant el següent:

    BUILD SUCCESSFUL in 1s
1 actionable task: 1 executed

    Com veiem, no mostra res, i això és bò. Si ara modifiquem el codi de la funció getGreeting() per a que retorne null en lloc de "Hello World!", tindrem una excepció com a resultat:

    > Task :app:test FAILED

AppTest > appHasAGreeting() FAILED
    org.opentest4j.AssertionFailedError at AppTest.java:12

1 test completed, 1 failed

FAILURE: Build failed with an exception.

* What went wrong:
Execution failed for task ':app:test'.
> There were failing tests. See the report at: file:///.../calculadora/app/build/reports/tests/test/index.html

* Try:
> Run with --scan to get full insights.

BUILD FAILED in 2s
3 actionable tasks: 2 executed, 1 up-to-date

    Com veiem, en la carpeta build/reports/tests/test tenim una pàgina web amb els resultats del test. Fes-li una ullada!

    Modificació del codi

    Una vegada hem vist les proves generades automàticament, anem a modificar el projecte. En primer lloc, canviarem l'estructura de directoris i els packages a com.ieseljust.edd, i crearem una nova classe Calculadora.java en els fonts i CalculadoraTest.java en els tests. Ens quedarà:

    app/src
├── main
│   ├── java
│   │   └── com
│   │       └── ieseljust
│   │           └── edd
│   │               ├── App.java
│   │               └── Calculadora.java
│   └── resources
└── test
    ├── java
    │   └── com
    │       └── ieseljust
    │           └── edd
    │               ├── AppTest.java
    │               └── CalculadoraTest.java
    └── resources

    Com podeu veure, no hem eliminat la classe App ni AppTest, encara que sí que caldrà canviar el nom dels packages a la nova estructura.

    El codi de la classe Calculadora serà el següent:

    package com.ieseljust.edd;

public class Calculadora {
    public int suma(int a, int b) {
        return a + b;
    }

    public int resta(int a, int b){
        return a-b;
    }
}

    I CalculadoraTest.java serà:

    package com.ieseljust.edd;

import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;

public class CalculadoraTest {

    private Calculadora calculadora;

    @BeforeEach
    public void setUp() {
        calculadora = new Calculadora();
    }

    @Test
    public void testSuma() {
        int resultat = calculadora.suma(3, 4);
        assertEquals(7, resultat, "La suma ha de ser correcta");
    }

    @Test
    public void testResta() {
        int resultat = calculadora.resta(7, 4);
        assertEquals(3, resultat, "La resta ha de ser correcta");
    }
}

    Com veiem, hem definit un parell de funcions de test: testSuma i testResta. Mirem abans alguns detalls més:

    • L'anotació @BeforeEach marca un mètide per a que s'execute abans de cada prova, i s'usa per configurar l'entorn de proves; és a dir, crear una instància calculadora.
    • Amb @Test ja hem vist que s'indiquen els mètodes de prova.
    • assertEquals és l'asserció que comprova si el valor esperat és igual al valor obtingut.

    Ara pots tornar a llançar el test i explorar els diferents resultats obtinguts. Prova a introduir algun error en els càlculs i comprova que dóna error.
