    2. Principis SOLID

    Principis SOLID

    Acoblament i cohesió

    Quan dissenyem programari, és fonamental que els components siguen modulars, reutilitzables i fàcils de mantenir. Per aconseguir-ho, hem de prestar especial atenció a dos conceptes clau:

    Acoblament

    L’acoblament (coupling) es refereix al grau de dependència entre diferents mòduls, classes o components del sistema.

    • Acoblament alt: Podem dir que existeix un alt acoblament:
    • Quan un canvi en una classe obliga a modificar moltes altres.

    • Quan les classes tenen moltes dependències entre elles.

      L'acoblament alt dificulta la reutilització i mantenibilitat del codi.

    • Acoblament baix(ideal): Podem dir que hi ha un acoblament baix:

    • Quan les classes tenen dependències mínimes entre si.
    • Quan cada classe se centra en una tasca específica.

    L'acoblament baix permet modificar una part del sistema sense afectar la resta.

    Exemple de codi Java amb alt acoblament (incorrecte)

    Definim les classe Database i UserService de la següent manera:

    class Database {
    public void connect() {
        System.out.println("Connecting to database...");
    }
}

class UserService {
    private Database db = new Database(); // Dependència directa

    public void getUser() {
        db.connect();
        System.out.println("Fetching user...");
    }
}

    Problema: La classe UserService depèn directament de Database. Si volem canviar la implementació de la base de dades, haurem de modificar el codi de UserService.

    Solució mitjançant acoblament baix (injecció de dependències):

    class Database {
    public void connect() {
        System.out.println("Connecting to database...");
    }
}

class UserService {
    private Database db;

    public UserService(Database db) { // Dependència injectada
        this.db = db;
    }

    public void getUser() {
        db.connect();
        System.out.println("Fetching user...");
    }
}

// Ús:
Database db = new Database();
UserService userService = new UserService(db);

    Avantatge: Ara podem substituir Database per una altra implementació sense modificar UserService.

    Injecció de dependències

    La injecció de dependències (Dependency Injection, DI) és un patró de disseny que consisteix a proporcionar les dependències d’una classe des de fora, en lloc que la pròpia classe cree les seues instàncies internament.

    Cohesió

    La cohesió (cohesion) mesura com de ben agrupades estan les responsabilitats dins d'una classe o mòdul.

    Un codi presenta una cohesió baixa quan una classe fa moltes coses diferents, la qual cosa dificulta la llegibilitat i la reutilització.

    És convenient tindre un codi que presenta una cohesió alta, és a dir, que una classe o mòdul tinga una responsabilitat clara i ben definida, de manera que facilite la modularitat i el manteniment.

    Exemple de codi amb cohesió baixa (incorrecte)

    Veiem la següent definició de classe en Java:

    class ReportGenerator {
    public void generateReport() {
        System.out.println("Generating report...");
    }

    public void sendEmail() {
        System.out.println("Sending report via email...");
    }

    public void saveToDatabase() {
        System.out.println("Saving report to database...");
    }
}

    Problema: Aquesta classe ReportGenerator té massa responsabilitats, no relacionades entre sí (crear informes, enviar correus i guardar dades).

    Solució millorada amb cohesió alta:

    class ReportGenerator {
    public void generateReport() {
        System.out.println("Generating report...");
    }
}

class EmailService {
    public void sendEmail() {
        System.out.println("Sending email...");
    }
}

class DatabaseService {
    public void saveToDatabase() {
        System.out.println("Saving to database...");
    }
}

    Amb això aconseguim ara que cada classe tinga una única responsabilitat, produint, a més, un codi més modular i mantenible.

    Cohesió i acoblament al Just Eat Pizza

    Retornem a l'exemple de la pizzeria de la unitat anterior. Recordem que disposem d'una API que retorna una llista de pizzes en format JSON. Suposem ara que el nostre mètode getAllPizzes() invoca directament a aquesta API i processa la resposta directament com un JSON.

    class PizzaService {
    async getAllPizzes() {
        const response = await fetch('https://pizza-rest-server-production.up.railway.app/api/pizzeria/pizzes');
        const jsonData = await response.json();
        return jsonData; // Retorna directament el JSON
    }
}

// Ús:
const service = new PizzaService();
service.getAllPizzes().then(data => console.log(data));

    Quin problema presenta aquesta situació?

    • Si l'API canvia i ara retornara XML en lloc de JSON, haurem de modificar el nostre codi per adaptar-lo.
    • El codi dependria directament del format de representació de les dades (JSON), la qual cosa provoca un acoblament fort.
    • Això afecta la reusabilitat i mantenibilitat, ja que qualsevol canvi en la API requereix modificacions en el codi.

    Quina solució podem proposar per reduir-ne l'acoblament?

    Per millorar aquest mal disseny, es crea una classe Pizza que represente una pizza com a objecte independentment del format de la resposta de l’API (tal i com està fet a l'exemple). Així, si la API canvia el seu format, només haurem de modificar el parser (la part que transforma o parseja un format a altre), però el nostre servei continuarà treballant amb objectes Pizza.

    export class Pizza extends Producte{
    constructor({id, nom, desc, vegetariana, alergens, img, preu}){
        // Propietats de Producte
        super({id, nom, img, preu})

        // Propietats addicionals de Pizza
        this.desc=desc;
        this.vegetariana=vegetariana;
        this.alergens=alergens;
    }
}



// Classe per processar diferents formats de resposta
class PizzaParser {
    static fromJson(jsonData) {
        return jsonData.map(pizza => 
            new Pizza(pizza)
        );
    }

    static fromXml(xmlData) {
        const parser = new DOMParser();
        const xmlDoc = parser.parseFromString(xmlData, "text/xml");
        const pizzas = [...xmlDoc.getElementsByTagName("pizza")];

        return pizzas.map(pizza => 
            new Pizza(
                pizza.getElementsByTagName("id")[0].textContent,
                pizza.getElementsByTagName("nom")[0].textContent,
                pizza.getElementsByTagName("desc")[0].textContent,
                ...
            )
        );
    }
}

// Servei de pizzes que usa el parser adequat
class PizzaService {
    async getAllPizzes(format = 'json') {
        const response = await fetch(`https://.../pizzes?format=${format}`);
        const data = await response.text();

        return format === 'json' 
            ? PizzaParser.fromJson(JSON.parse(data)) 
            : PizzaParser.fromXml(data);
    }
}

// Ús:
const service = new PizzaService();
service.getAllPizzes('json').then(pizzes => console.log(pizzes));
service.getAllPizzes('xml').then(pizzes => console.log(pizzes));

    Amb aquest codi aconseguim reduir l'acoblament, ja que PizzaService no depèn directament del format de l'API, sinó d’una abstracció (PizzaParser).

    A més, hem aconseguit una alta cohesió al separar la lògica de conversió (PizzaParser) de la lògica de servei (PizzaService).

    Per altra banda, també hem generat un codi més mantenible, ja que si ara l'API afegira un nou format (per exemple, CSV), només caldria afegir un nou mètode a PizzaParser sense modificar PizzaService.

    Principis SOLID

    Els principis SOLID ajuden a crear un programari més flexible i mantenible. Aquests principis van ser definits per Robert C. Martin (Uncle Bob) i són fonamentals en la programació orientada a objectes. Anem amb ells.

    Principi 1. S - Principi de responsabilitat única (Single Responsibility Principle - SRP)

    Cada classe ha de tenir una única responsabilitat i una sola raó per canviar.

    Si una classe fa massa coses, es torna difícil de mantenir i modificar. Separant les responsabilitats, millorem la llegibilitat i facilitem la reutilització.

    • Exemple incorrecte (classe amb múltiples responsabilitats):
    class Order {
    public void calculateTotal() { /* càlcul de total */ }
    public void printInvoice() { /* impressió de factura */ }
    public void saveToDatabase() { /* desar en BD */ }
}
    • Exemple Correcte (separem responsabilitats en classes diferents):
    class Order {
    public void calculateTotal() { /* càlcul de total */ }
}

class InvoicePrinter {
    public void printInvoice() { /* impressió de factura */ }
}

class OrderRepository {
    public void saveToDatabase() { /* desar en BD */ }
}

    Principi 2. O - Principi d'Obert-Tancat (Open/Closed Principle - OCP)

    Les classes han d'estar obertes a l'extensió però tancades a la modificació.

    Aquest principi ens indica que el codi ha de permetre afegir funcionalitats noves (obert a l'extensió) sense haver de modificar les classes existents (tancades a la modificació), reduint el risc d'introduir errors en el sistema.

    • Exemple incorrecte: El següent codi presenta una classe ShapePrinter que imprimeix diferents formes (Rectangle, Circle).
    class Rectangle {
    public void draw() { System.out.println("Dibuixant rectangle"); }
}

class Circle {
    public void draw() { System.out.println("Dibuixant cercle"); }
}

class ShapePrinter {
    public void print(Rectangle rect) { rect.draw(); }
    public void print(Circle circ) { circ.draw(); }
}

    El problema que presenta aquest codi és que, cada vegada que volem afegir una nova forma (per exemple, Triangle o Hexagon), hem de modificar ShapePrinter, cosa que trenca el principi d’Obert-Tancat: hem de modificar la classe ShapePrinter per poder afegir-li noves funcionalitats, obrint-la per tant a la modificació per estendre-la. Això fa que la solució siga difícil d'escalar i de mantenir.

    • Exemple Correcte: Per corregir la situació anterior, podem fer que ShapePrinter funcione amb una interfície o classe abstracta Shape, de manera que qualsevol nova forma només haja d’implementar Shape i ShapePrinter no s'haja de modificar.
    // Es defineix com a classe abstracta, però podria ser també una interfície
abstract class Shape {
    public abstract void draw();
}

class Rectangle extends Shape {
    public void draw() { System.out.println("Dibuixant rectangle"); }
}

class Circle extends Shape {
    public void draw() { System.out.println("Dibuixant cercle"); }
}

class ShapePrinter {
    public void print(Shape shape) { shape.draw(); }
}

    Si ara volem crear una nova classe Triangle, no modificariem res de l'anterior:

    class Triengle extends Shape {
    public void draw() { System.out.println("Dibuixant Triangle"); }
}

    Principi 3. L - Principi de substitució de Liskov (Liskov Substitution Principle - LSP)

    Les subclasses han de poder substituir les seves superclasses sense trencar el comportament del programa.

    Si una subclasse altera el comportament esperat de la classe base, pot provocar errors inesperats en altres parts del sistema. És a dir, si una classe A és la superclasse d'una classe B, qualsevol instància de B ha de comportar-se com una instància de A sense trencar el codi.

    • Exemple Incorrecte (una subclasse que no manté el comportament de la superclasse):
    class Bird {
    public void fly() { System.out.println("Flying"); }
}

class Penguin extends Bird {
    // Els pingüins no poden volar!
}

    En aquest cas, la classe Penguin trenca el comportament de la classe base, ja que no pot tindre un mètode fly.

    • Correcció: Hem de redefinim l'estructura per evitar el problema. Com que no totes les aus volen, deixem la classe Bird com a classe abstracta i creem una nova classe per a les aus que sí que volen:
    abstract class Bird { }

class FlyingBird extends Bird {
    public void fly() { System.out.println("Flying"); }
}

class Penguin extends Bird { }

    Un exemple sobre la pizzeria

    A l'exemple de la pizzeria, quan gestionem les comandes amb llistes de productes, estem aplicant el Principi de Substitució de Liskov (LSP) de manera correcta, sempre que les subclasses (Pizza, Entrant, Beguda) puguen substituir la classe base (Producte) sense trencar el codi.

    Com podem vore, aquest principi està estretament relacionat amb el polimorfisme en la POO. El polimorfisme permet que un objecte d'una subclasse siga tractat com un objecte de la seua superclasse, cosa que encaixa perfectament amb el principi LSP.

    • El polimorfisme ens permet escriure codi genèric i reutilitzable:
    • Si una subclasse pot substituir la seva superclasse sense problemes, podem fer servir el polimorfisme per evitar el codi depenent de tipus concrets.
    • Això fa que el codi siga més flexible i extensible sense necessitat de modificar-lo.
    • El principi LSP garanteix que el polimorfisme funcione correctament:
    • Si una subclasse no es comporta com la superclasse esperada, el polimorfisme pot trencar-se.
    • LSP assegura que, si fem servir una subclasse en lloc de la superclasse, el codi continuarà funcionant sense modificar-se.

    Principi 4. I - Principi de segregació d'interfícies (Interface Segregation Principle - ISP)

    És millor tenir interfícies menudes i específiques que una única interfície gran.

    Segons aquest principi, si una interfície obliga una classe a implementar mètodes que no necessita, significa que està mal dissenyada. És preferible dividir-la en interfícies més menudes i enfocades. Veiem-ho amb un exemple:

    • Exemple Incorrecte (interfície massa gran): La següent interfície implementa els mètodes work i eat.
    interface Worker {
    void work();
    void eat();
}

    Això obliba a que totes les classes que la implementen, implementen ambdós mètodes. Si tenim una classe que necessita implementar work però no eat, estem obligant-la a implementar els dos.

    • Exemple Correcte (interfícies més específiques): Per donar solució, segreguem aquesta interfície en dues:
    interface Workable {
    void work();
}

interface Eatable {
    void eat();
}

    De manera que una classe pot implementar una, altra o les dues alhora, segons les seues necessitats.

    Principi 5. D - Principi d'inversió de dependències (Dependency Inversion Principle - DIP)**

    Les dependències han d'estar en abstraccions, no en classes concretes.

    Aquest principi significa que les classes d'alt nivell no han de dependre de les de baix nivell. En canvi, totes han de dependre d'abstraccions (interfícies o classe abstractes) per fer el codi més flexible i fàcil d’adaptar. Veiem-ho amb un exemple:

    • Problema: Violació del principi DIP

    Imaginem que tenim una classe ComandaService que depèn directament d’una implementació específica de Database per a guardar comandes en la base de dades.

    class MySQLDatabase {
    public void saveOrder(String order) {
        System.out.println("Desant la comanda en MySQL: " + order);
    }
}

class ComandaService {
    private MySQLDatabase database = new MySQLDatabase(); // DEPENDÈNCIA DIRECTA!

    public void save(String order) {
        database.saveOrder(order);
    }
}

    Aquesta és una mala solució pels següents motius:

    • Presenta un acoblament fort, ja que ComandaService depèn directament de MySQLDatabase. Si volem canviar a una altra base de dades (per exemple, PostgreSQL), hem de modificar ComandaService.
    • Presenta poca flexibilitat, ja que no podem reutilitzar ComandaService amb una altra implementació de base de dades sense modificar el codi existent.
    • Dificultat les proves: Si volem fer tests de ComandaService, necessitem una instància real de MySQLDatabase, la qual cosa complica les proves unitàries.

    Com ajuda l'aplicació del principi DIP?

    Podem resoldre aquest problema introduint una interfície Database, que encapsule el comportament esperat de qualsevol base de dades. Així, ComandaService treballarà amb la interfície i no amb una implementació concreta.

    // Interfície genèrica per a la base de dades
interface Database {
    void saveOrder(String order);
}

// Implementació per a MySQL
class MySQLDatabase implements Database {
    public void saveOrder(String order) {
        System.out.println("Guardant la comanda en MySQL: " + order);
    }
}

// Implementació per a PostgreSQL
class PostgreSQLDatabase implements Database {
    public void saveOrder(String order) {
        System.out.println("Guardant la comanda en PostgreSQL: " + order);
    }
}

// Ara ComandaService depèn d'una ABSTRACCIÓ (Database), no d'una implementació concreta
class ComandaService {
    private Database database;

    public ComandaService(Database database) { // Injecció de dependències
        this.database = database;
    }

    public void save(String order) {
        database.saveOrder(order);
    }
}

// Ús del servei amb diferents bases de dades
public class Main {
    public static void main(String[] args) {
        Database mysql = new MySQLDatabase();
        Database postgres = new PostgreSQLDatabase();

        ComandaService comanda1 = new ComandaService(mysql);
        comanda1.save("Pizza 4 formatges");

        ComandaService comanda2 = new ComandaService(postgres);
        comanda2.save("Pizza Barbacoa");
    }
}

    Quins beneficis aporta aquesta solució?

    • Baix acoblament: ComandaService no sap ni li importa quina base de dades s’està utilitzant.
    • Flexibilitat: Podem afegir noves bases de dades (MongoDB, Firebase, etc.) sense modificar ComandaService.
    • Més fàcil de provar: Podem crear un MockDatabase per a fer proves unitàries sense utilitzar una base de dades real.
