    4.1. Patrons creacionals

    Patrons Creacionals

    Els patrons creacionals s'encarreguen de gestionar la creació d'objectes, i permeten decidir el moment i la forma en què es creen aquests (en altres paraules, quan, on i com es fa el new per crear l'objecte).

    Aquests patrons ens asseguren sistemes més fàcils de manteniir i més escalables, ja que aïllen la lògica de creació dels objectes i ens proporcionen flexibilitat per canviar la forma en què es crearan els objectes.

    Refactoring Guru

    El lloc web Refactoring.guru és tot un referent en quant a patrons de disseny, patrons de refactorització i principis SOLID.

    En ell s'hi descriuren una gran quantitat de patrons, i podeu veure la seua implementació en diversos llenguatges de programació.

    En aquesta unitat ens basarem en moltes de les explicacions que hi fa el lloc web, però allí disposes de molta més informació per ampliar els teus coneixements.

    Classificació dels Patrons Creacionals

    Segons Refactoring Guru, els principals patrons creacionals són:

    1. Factory Method: Permet crear objectes d'una classe sense especificar-ne la classe exacta.
    2. Abstract Factory: Proporciona una interfície per crear famílies d'objectes sense especificar les classes concretes.
    3. Singleton: Garanteix que només existisca una instància d’una classe en tota l'aplicació.
    4. Builder: Permet crear objectes complexos pas a pas.
    5. Prototype: Crea nous objectes clonant un prototip.

    Anem a veure alguns d'aquests exemples, però recordeu que teniu més informació a Refactoring.guru

    1. Factory Method

    El Factory Method és un patró que permet crear objectes de manera flexible i sense especificar la classe exacta de l'objecte a crear. Aquest patró defineix una interfície per crear un objecte, però deixa que les subclasses decidisquen quina classe instanciar.

    Exemple de codi: Enviament de comandes

    Imaginem que en el sistema de la pizzeria, quan el client fa una comanda, pot triar entre recollir-la a la pizzeria o entre enviament a domicili. Cada tipus d'enviament té diferents característiques i maneres de gestionar-se.

    En aquest cas, ens centrem en el desenvolupament en la part del servidor, que estaria implementat, per exemple amb Java.

    El patró Factory Method ens permetria crear el servei d'enviament adequat sense haver de saber exactament quina implementació concreta utilitzar (recollida o enviament a domicili). Això facilita l'extensió del sistema per afegir més opcions d'enviament més endavant (per exemple, enviament exprés, lliurament amb dron, etc.), tot amb un codi net i mantenible.

    Com podríem aplicar Factory Method?

    Cada tipus d’enviament podria ser tractat com un tipus d’objecte diferent, i la fàbrica decidiria quin tipus crear en funció de la selecció del client.

    Per a això, hem de realitzar les següents tasques:

    1. Crear interfície (classe abstracta) comú: Tots els tipus d’enviament heretarien d’una classe base comuna Enviament amb mètodes comuns, com per exemple calcularCost() i processar().

    2. Crear fàbriques concretes: Tenim una fàbrica abstracta EnviamentFactory, i les fàbriques concretes (EnviamentDomiciliFactory i RecollidaFactory) crearien els objectes específics per a cada tipus d'enviament.

    Veiem com quedaria el codi de manera simplificada:

    // Interfície comuna per a tots els tipus d'enviament
abstract class Enviament {
    // Suposem que ja tenim la classe Producte definida prèviament
    protected List<Producte> productes = new ArrayList<>();
    public abstract void processar();
    public abstract double calcularCost();
}

// Tipus d'enviament: A domicili
class EnviamentDomicili extends Enviament {
    private double preu;

    public EnviamentDomicili() {
        // Preu fix inicial per a l'enviament a domicili
        // (l'enviament a domicili té un cost de 3€)
        this.preu = 3.00; 
    }

    @Override
    public void processar() {
        System.out.println("Enviant la comanda a domicili...");
    }

    @Override
    public double calcularCost() {
        double totalProductes = 0;
        for (Producte p : productes) {
            totalProductes += p.getPreu();
        }
        return totalProductes + this.preu;  // Cost dels productes + cost d'enviament
    }
    }
}

// Tipus d'enviament: Recollida en tenda
class RecollidaTenda extends Enviament {
    @Override
    public void processar() {
        System.out.println("La comanda està llesta per a ser recollida a la tenda.");
    }

    @Override
    public double calcularCost() {
        double totalProductes = 0;
        for (Producte p : productes) {
            totalProductes += p.getPreu();
        }
        return totalProductes;  // Només el cost dels productes (sense cost d'enviament)
    }
}

/*
Ara definim les diferents fàbriques per crear els objectes
*/

// Fàbrica abstracta per crear els enviaments 
abstract class EnviamentFactory {
    public abstract Enviament createEnviament();
}

// Fàbrica per a enviament a domicili
class EnviamentDomiciliFactory extends EnviamentFactory {
    @Override
    public Enviament createEnviament() {
        return new EnviamentDomicili();
    }
}

// Fàbrica per a recollida en tenda
class RecollidaFactory extends EnviamentFactory {
    @Override
    public Enviament createEnviament() {
        return new RecollidaTenda();
    }
}

    Què podem extreure del codi anterior?

    • Enviament és la classe base amb dos mètodes: processar(), per iniciar el procés d'enviament o recollids, i calcularCost(),per calcular el cost de l'enviament.
    • Les classes concretes EnviamentDomicili i RecollidaTenda implementen aquests mètodes de manera diferent, seguint el patró Factory Method.
    • Les fàbriques EnviamentDomiciliFactory i RecollidaFactory decideixen quina classe concreta crear, depenent de la selecció de l'usuari.

    Una vegada definit aixó, la classe principal utilitzaria aquestes fàbriques de la forma següent:

    // Selecció del tipus d'enviament, per exemple, enviament a domicili
// Observeu que definim factory com a la classe base per a les fàbriques (EnviamentFactory)
// però el que creem és una subclasse d'aquesta (EnviamentDomiciliFactory)
EnviamentFactory factory = new EnviamentDomiciliFactory();

// Ara creem enviament, definint-la com a la classe base,
// Però la fàbrica ens retornarà la subclasse (concreta) corresponent
Enviament enviament = factory.createEnviament();

// Ara podem processar la comanda (això seria independent del tipus)
enviament.processar();  

// I calcular el seu cost
System.out.println("Cost total de la comanda: " + enviament.calcularCost() + "€");


// Per a una recollida en tenda faríem
factory = new RecollidaFactory();
Enviament recollida = factory.createEnviament();
recollida.processar();  // Processem la comanda de recollida
System.out.println("Cost total de la comanda: " + enviament.calcularCost() + "€");

    2. Singleton

    El patró Singleton assegura que només hi haurà una instància d’una classe durant tota l'execució del programa i proporciona un punt global d'accés a aquesta mateixa instància.

    Això ens és de gran utilitat en tots aquells casos que necessitem tindre, per exemple una instància única d'una connexió a una base de dades, o un accés únic a un Repositori d'informació.

    Podem veure el funcionament en el següent diagrama de classes:

    Exemple

    Suposem que volem gestionar la connexió a la base de dades de manera que només hi haja una instància de la connexió durant tota l'aplicació.

    class DatabaseConnection {
    // Definim una instància de la pròpia classe de manera interna
    private static DatabaseConnection instance;

    private DatabaseConnection() {
        // Constructor privat per evitar la creació d'instàncies externes
        // Com que és privat... ningú por fer new des de fora!
    }

    public static DatabaseConnection getInstance() {
        // Creem un mètode estàtic específic per obtenir la 
        // instància de la classe


        if (instance == null) {
            // Si la instància no existeix, es crea.
            // Aquesta és á única vegada que es farà new.
            instance = new DatabaseConnection();
        }

        // I retornem la instància
        return instance;
    }

    public void connect() {
        System.out.println("Conectant a la base de dades...");
    }
}

    Veiem què hem fet:

    • Instància única: La classe DatabaseConnection inclou una referència a ella mateixa, que serà única, i només serà instanciada una vegda mitjançant el mètode estàtic getInstance().
      • Constructor privat i mètodes estàtics: El fet que el constructor siga privat implica que ningú, des de fora podrà crear instàacies. La classe sofereix el mètode estàtic getInstance() per retornar, de forma controlada l'accés a la classe.
    • El mètode estàtic getInstance() ens assegura que només es crea una vegada la instància de la classe. Quan s'invoca, aquest comprova si la referència és nul·la. Si la instància no és nul·la, la recorna, i en cas que siga nul·la, la crea (i després la retorna).
    • Accés global: Això garanteix que en qualsevol punt del codi, s'accedeixi sempre a la mateixa instància de la connexió a la base de dades.
