    4.3.patrons estructurals

    Patrons Estructurals

    Els patrons estructurals s'ocupen de com organitzar i compondre les classes i els objectes en sistemes complexos. Aquests patrons ajuden a simplificar l'estructura de classes i reutilitzar el codi mitjançant la creació d'estructures més eficients i modulars. Els patrons estructurals són especialment útils quan tenim una arquitectura complexa o quan necessitem adaptar components ja existents.

    Els principals patrons estructurals que es comenten a Refactoring Guru són:

    • Adapter
    • Bridge
    • Composite
    • Decorator
    • Facade
    • Flyweight
    • Proxy

    D'entre ells anem a centrar-nos en un parell de patorns, i veurem com es poden aplicar en situacions comunes.

    • Adapter, que ens permetrà que dues classes incompatibles treballen jnuntes, a través d'una interfície comuna,
    • Decorator: Afegeix funcionalitat addicional a un objecte de manera flexible sense modificar la seva estructura.

    En aquest apartat ens centrarem en Adapter i Decorator, explicant com es poden aplicar en situacions comunes, com en l'exemple de tractament de fluxos a Java.

    Patró Adapter

    El patró Adapter s'utilitza quan necessitem que dues classes que no són compatibles treballin conjuntament. Aquest patró crea una interfície comuna entre les dues classes que tenen interfícies incompatibles, i adapta una classe perquè puga ser utilitzada per l’altra.

    El patró Adapter és una solució ideal per als tipus de casos en què necessitem adaptar representacions de dades en formats diferents. Per exemple, podriem crear un Adaptador que convertisca dades en format XML que obtinim d'una API a format JSON perqupe la nostra aplicació treballe amb ells.

    Una altra situació on ens trobem aquest patrons és en el desenvolupament d'aplicaicons en Android, on per crear desplegables o llistes a partir d'informació variada (un XML, un vector...), necessitem un Adaptador que agafe la informació en el format original i l'adapte o genere els components corresponents (items del desplegable o la llista).

    Exemple pràctic: Una API per a les pizzes en XML

    Imaginem que l'API de la pizzeria, en lloc de retornar-nos la informació en JSON ho fa en XML. La nostra aplicació espera rebre-les en JSON, pel que necessitarem fer ús d'aquest patró per adaptar l'XML a JSON.

    Per a això, farem ús d'una URL que ens proporcione la informació de les pizzes en format XML pur.

    Projecte d'exemple

    Deixarem l'exemple que anem treballant al repositori del projecte de la pizzeria (webapp_eljust_eat_pizza), concretament, dins la carpeta Sprint5_Adapters.

    El patró Repository

    El patró Respository no és un patró de disseny com a tal (per això no el voreu a Refactoring Guru), sinò un patró arquitectònic. Es tracta d'una abstracció per gestionar l'accés a les dades, que es fa servir per millorar l'escalabilitat i tindre un codi més mantenible.

    Comentem aquest patró repository perquè anem a introduir aquesta nova capa al nostre exemple de la pizzeria abans d'abordar el patró Adapter.

    La nostra aplicació, arquitectònicament s'estructurava en les següents capes:

    • components : amb els webcomponents, que són les vistes,
    • libcomponents: amb la lliberria base de components,
    • models: Amb els models del domini de l'aplicació (pizzes, etc)
    • services: Amb la funcionalitat d'accés al servei
    • state: Amb l'estat de l'aplicació (classe carret)

    Al nostre codi, des del mateix main i les diferents vistes (webComponents), hem estat accedint directament a la capa de servei, suposant que aquesta ja ens retorna la representació en JSON esperada.

    Si ara canviarem la forma de representació de les dades, i haverem de modificar aquest servei per un altre, hauriem de modificar totes les referències a aquest servei que tinguerem pel codi.

    Per afegir un major nivell d'abstracció i desentendre'ns del servei que hi ha per baix, afegim una nova capa de repositori, que centraritzarà la informació que necessita la nostra aplicació, de manera independent a la capa de serveis.

    Inicialment, aquesta capa serà un pont entre la nostra aplicació i la pròpia capa de servei.

    Aquesta capa contindrà una classe PizzeriaRepository amb el següent codi (observeu que té els mateixos mètodes que PizzaService, i que l'únic que fa és retornar el que retornen estos):

    import {PizzeriaService} from '../services/pizzeriaService.js';

export class PizzeriaRepository {
    static async getAllPizzes() {
        return PizzeriaService.getAllPizzes();
    }

    static async getAllEntrants() {
        return PizzeriaService.getAllEntrants();
    }

    static async getAllDrinks() {
        return PizzeriaService.getAllDrinks();
    }

    static async sendOrder(comanda) {
        return PizzeriaService.sendOrder(comanda);

    }
}

    Per utlitzar aquest repository, el que gem és reemplazar totes les referències a PizzeriaService de la resta de codi per referències a PizzeriaRepository

    Una vegada tenim clara l'explicació anterior, anem a implementar un nou mètode en el servei, que accedisca i ens retorne una representació en XML de les dades.

    Codi d'exemple:

    1. API que retorna dades en XML: L'API de la pizzeria retorna una llista de pizzes en XML, i nosaltres volem convertir aquesta resposta a JSON per poder treballar amb ella.

    2. Adapter per convertir XML a JSON: Crearem un adaptador per convertir l'XML a JSON.

    Implementació del patró Adapter:

    import org.w3c.dom.*;
import javax.xml.parsers.*;
import java.io.*;
import org.json.JSONObject;

// Classe base per representar una pizza (format JSON)
class Pizza {
    private String nom;
    private double preu;

    public Pizza(String nom, double preu) {
        this.nom = nom;
        this.preu = preu;
    }

    public String getNom() {
        return nom;
    }

    public double getPreu() {
        return preu;
    }
}

// Adaptador per convertir XML a JSON
interface PizzaAdapter {
    JSONObject adaptarAJSON(String xml);
}

class PizzaXMLAdapter implements PizzaAdapter {
    @Override
    public JSONObject adaptarAJSON(String xml) {
        try {
            // Convertir XML a JSONObject
            DocumentBuilderFactory factory = DocumentBuilderFactory.newInstance();
            DocumentBuilder builder = factory.newDocumentBuilder();
            InputStream inputStream = new ByteArrayInputStream(xml.getBytes());
            Document doc = builder.parse(inputStream);
            doc.getDocumentElement().normalize();

            // Suposant que l'XML té una estructura similar a:
            // <pizza>
            //    <nom>Margarita</nom>
            //    <preu>8.5</preu>
            // </pizza>
            NodeList nodeList = doc.getElementsByTagName("pizza");
            JSONObject pizzaJSON = new JSONObject();
            for (int i = 0; i < nodeList.getLength(); i++) {
                Node node = nodeList.item(i);
                if (node.getNodeType() == Node.ELEMENT_NODE) {
                    Element element = (Element) node;
                    String nom = element.getElementsByTagName("nom").item(0).getTextContent();
                    double preu = Double.parseDouble(element.getElementsByTagName("preu").item(0).getTextContent());

                    // Creem el JSON a partir de l'XML
                    pizzaJSON.put("nom", nom);
                    pizzaJSON.put("preu", preu);
                }
            }
            return pizzaJSON;
        } catch (Exception e) {
            e.printStackTrace();
        }
        return null;
    }
}

// Exemple d'ús: convertim un XML a JSON
public class Main {
    public static void main(String[] args) {
        // Exemples d'XML que retorna l'API
        String xml = "<pizza><nom>Margarita</nom><preu>8.5</preu></pizza>";

        // Utilitzant el adaptador
        PizzaAdapter adapter = new PizzaXMLAdapter();
        JSONObject pizzaJSON = adapter.adaptarAJSON(xml);

        // Mostra el resultat en JSON
        System.out.println(pizzaJSON.toString());
    }
}

    Explicació del patró Adapter en aquest exemple:

    • PizzaAdapter: És una interfície que defineix el mètode adaptarAJSON(), que és el que permetrà adaptar l'XML al format JSON.
    • PizzaXMLAdapter: És l'adaptador concret que converteix l'XML rebut de l'API a JSON. Aquest adaptador processa l'XML i crea un objecte JSONObject que l'aplicació pot utilitzar.
    • Pizza: Aquesta classe representa la pizza amb els atributs nom i preu. Tot i que no s'utilitza explícitament a l'exemple, el JSON resultant de l'adaptador podria ser usat per crear instàncies de Pizza.

    Com funciona l'adaptador?

    • L'API original retorna les dades de les pizzes en format XML, però la nostra aplicació espera un objecte en format JSON.
    • El PizzaXMLAdapter adapta les dades XML a una estructura JSON, permetent que la nostra aplicació treballi amb el format esperat.
    • El codi fa servir el mètode adapatarAJSON per convertir l'XML en un JSONObject que conté la informació de la pizza.

    Conclusió:

    El patró Adapter és molt útil en situacions on necessitem convertir entre formats incompatibles, com en aquest cas, convertir dades d'una API XML a un format que la nostra aplicació pot manejar (JSON). El patró permet que la nostra aplicació sigui flexible i adaptable a canvis de format sense modificar el codi existent.

    2. Patró Decorator

    El patró Decorator s’utilitza per afegir funcionalitat a un objecte de manera dinàmica. El principal avantatge d’aquest patró és que ens permet modificar el comportament d’un objecte sense modificar la seva classe base, cosa que el fa extremadament flexible.

    Exemple de codi (Decorator):

    Un exemple clàssic del patró Decorator es troba en el tractament de fluxos de dades a Java. Les classes InputStream i OutputStream utilitzen decoradors per afegir funcionalitat extra, com la lectura de dades en un format determinat (per exemple, llegir dades comprimides o encriptades).

    import java.io.*;

// Component base
class Coffee {
    public String serve() {
        return "Cafè";
    }
}

// Decorador base
abstract class CoffeeDecorator extends Coffee {
    protected Coffee decoratedCoffee;

    public CoffeeDecorator(Coffee coffee) {
        this.decoratedCoffee = coffee;
    }

    public String serve() {
        return decoratedCoffee.serve();
    }
}

// Decorador concret 1
class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee coffee) {
        super(coffee);
    }

    @Override
    public String serve() {
        return decoratedCoffee.serve() + " amb llet";
    }
}

// Decorador concret 2
class SugarDecorator extends CoffeeDecorator {
    public SugarDecorator(Coffee coffee) {
        super(coffee);
    }

    @Override
    public String serve() {
        return decoratedCoffee.serve() + " amb sucre";
    }
}

// Exemple d'ús
public class Main {
    public static void main(String[] args) {
        Coffee coffee = new Coffee();
        System.out.println(coffee.serve()); // Cafè

        coffee = new MilkDecorator(coffee);
        System.out.println(coffee.serve()); // Cafè amb llet

        coffee = new SugarDecorator(coffee);
        System.out.println(coffee.serve()); // Cafè amb llet amb sucre
    }
}

    Explicació del patró Decorator:

    • Coffee és el component base, que serveix un cafè senzill.
    • CoffeeDecorator és una classe abstracta que permet afegir funcionalitat extra a qualsevol objecte Coffee de manera dinàmica.
    • MilkDecorator i SugarDecorator són decoradors concrets que afegeixen la llet i el sucre, respectivament.
    • El comportament de serve() es modifica dinàmicament sense haver de modificar el comportament original de la classe Coffee.

    Conclusió:

    Els patrons estructurals com Adapter i Decorator són extremadament útils per millorar la flexibilitat del codi i permetre la modificació de comportaments sense tocar el codi original.

    • Adapter és ideal quan tens classes incompatibles i necessites adaptar-les per poder treballar amb elles de manera conjunta.
    • Decorator és perfecte quan necessites afegir funcionalitats a un objecte sense canviar la seva classe base, com en el cas de la manipulació de fluxos de dades a Java.
